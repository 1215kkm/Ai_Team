# 강체크 새 규칙 — 앱 백엔드 "보안 감사"를 QA 검수에 필수 포함

- **출처 프로젝트**: `km-cafe24` (카페24 스킨 + 주노비스 앱)
- **일자**: 2026-06-25
- **발견 경위**: 강체크(QA)·강개발 검수를 **스킨/프론트만** 범위로 의뢰함 → 앱 백엔드(PHP API)의 인증 결함을 아무도 안 봄. `settings.php`가 무인증 쓰기를 허용해 **누구나 `?mall=타몰` 로 남의 몰 설정·상품가격 변조 + 고객 연락처(PII) 조회** 가능한 P0 홀이 라이브에 존재. **CEO 질문("a구매자가 b구매자 설정 못 건드리지?")으로 발견.**

## 핵심 교훈
검수 에이전트가 "놓친" 게 아니라 **PM/오케스트레이터가 검수 범위에 백엔드 보안을 안 넣은 것**. 돈·개인정보·멀티테넌트(몰별) 영향이 있는 백엔드가 있으면, 프론트 검수와 **별개로 보안 감사를 명시 의뢰**해야 한다.

## 재사용 가능 규칙 (강체크 = qa 에이전트)
돈/개인정보/타테넌트에 영향 주는 **백엔드 엔드포인트가 있는 모든 제품**은 QA 검수 시 아래 보안 감사를 별도 수행한다:

1. **모든 쓰기(write) 엔드포인트가 운영자 인증을 요구하는가?** (`mall`/tenant 파라미터를 무조건 신뢰하지 말 것 — 위조 가능)
2. **PII(개인정보) 조회/노출 엔드포인트가 인증으로 보호되는가?**
3. **CORS `*` 엔드포인트가 민감한 쓰기/읽기를 허용하지 않는가?**
4. **debug/진단 파라미터가 내부상태(토큰·API오류·원본데이터)를 무인증 노출하지 않는가?**
5. **멀티테넌트 격리**: A 사용자가 tenant id를 바꿔 B의 데이터를 건드릴 수 없는가?
6. **클라이언트측 플래그는 보안 경계가 아니다** (`?wnadmin=1`, localStorage, 숨김 UI 등) — 반드시 서버에서 검증.
7. 인증 토큰은 **추측 금지** — 플랫폼 공식 샘플의 검증식을 확인하고 구현 (예: 카페24 앱 실행 HMAC = base64(HMAC-SHA256(쿼리스트링에서 마지막 &hmac= 제거분, client_secret))).
